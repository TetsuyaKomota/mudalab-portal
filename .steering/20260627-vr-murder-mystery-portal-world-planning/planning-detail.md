# VRマダミスポータルワールド 企画詳細

## 企画概要

### 企画名
- VRマダミスポータルワールド

### 目的
- VRChat上で複数のVRマダミス作品を選び，集まった人数に応じて遊べるポータルワールドを作る
- 前作「ぼくらのペリン事件」で得たGMレス進行，アイテム公開・譲渡，観戦，ハンドアウト管理の知見を共通基盤化する
- 作品ごとのUnity実装負荷を下げ，脚本・作品データ・アイテム素材の分担制作を可能にする

### 成果物の方向性
- 最終成果物は発表用の単一HTMLスライド
- 本資料はスライド作成前の企画詳細資料
- 想定読者は制作チーム

## 背景

### 前作概要
- 「ぼくらのペリン事件」はVRChat上のGMレスVRマダミス風ゲームワールド
- 4人プレイ専用
- 観戦対応
- プレイ時間は約240分
- 日常推理系，死者なし
- ロールプレイより推理・謎解き重視
- 真実の情報はアイテムとして提示し，口頭会話は嘘も可能
- 個人目標とエンディング分岐あり

### 次回作で改善したい前作制約
- 4人集まらないと遊べない
- 作品固有ロジックが多く，再利用しづらい
- 脚本・実装・アイテム制作が分離しづらい
- 長時間プレイ前提で心理的ハードルが高い
- テストプレイに必要人数と時間がかかる

## 前作振り返りKPT

### Keep
- ユーザーステート管理
    - キャラクター，所持情報，進行状況を管理する仕組みは有効
- GMレス進行
    - 個別ハンドアウトを作り込むことでGMレス進行が成立
- 密談システム
    - VR空間で自然に会話を分けられる点が面白い
- アイテム公開・譲渡
    - 情報を物体として扱い，見せる・渡す体験が有効
- ローカルQvPen
    - 推理メモや議論補助として有効
    - OSSとして活用し，自前実装は不要
- 観客システム
    - 発想として有効
    - プレイ後に観客も感想戦へ入りやすい導線は継承
- ハンドアウトとアイテムリスポーン
    - 情報・アイテムを失っても復帰できる安心感がある
- 全員同意によるフェーズ移行
    - 参加者全員がボタンを押して進行する納得感がある

### Problem
- 制作に時間がかかりすぎた
    - ストーリーを考えすぎた
    - 複数事件・複数フェーズの整合性確認が重かった
- 実装の再利用性が低い
    - 複雑な実装でバグ取りに時間がかかった
    - フェーズ・投票・アイテム・エンディングが密結合
    - 作品固有ロジックが多く汎化しづらい
- VRChat同期・所有権まわりの検証負荷が高い
- テストプレイの仕組みが弱い
    - 必要人数を集めづらい
    - 長時間テストが必要
- ハンドアウト携帯の負荷
    - 特にデスクトップモードで情報確認が重い
- 観客システムが大味
    - 声の聞き分け機能は未実装
    - アイテム確認はクリア部屋との併用
- 制作分担が難しい
    - ワールド開発，脚本制作，アイテムモデリングをきれいに分けたい
- プレイ時間が長く，遊ぶ心理的ハードルが高い

### Try
- カセットシステム
    - 作品をカセット単位で選択・セット
    - カセットに人数，所要時間，難易度，ネタバレ設定を持たせる
    - カセット交換でワールド共通基盤に別作品データを読み込む
    - 複雑な作品は専用ワールドへのポータルとして扱う
- 脚本のGit分離
    - 作品データをリポジトリ管理
    - PRマージで作品追加候補に入る運用
    - ハンドアウトを事前にリポジトリで読める構成
    - JSON / Markdownなど人が編集しやすい形式を検討
- マッチング
    - Publicで募集部屋を立て，人数が揃ったら開始
    - ワールド内に複数テーブルを配置
    - テーブルごとにカセット選択と参加者決定を行う
    - 待機中に作品概要・所要時間・募集人数を見られる
    - 観戦待ち，参加待ち，感想戦を同じ導線に置く
    - 途中離脱や人数不足時の中止・再募集フローを定義
- 人数可変
    - 作品ごとに `minPlayers` / `maxPlayers` / `recommendedPlayers` を持たせる
    - 2人短編，3-5人可変，4人固定などを同じポータル内に並べる
    - 参加者数はカセットに依存
- ユーザー制作
    - 初期MVPでは非常にシンプルなデータ差し替えに限定
    - フェーズ構成は固定
    - 専用ギミック追加は初期MVPから外す
    - 制作者向けテンプレートと検証チェックリストを用意
- ハンドアウト改善
    - いつでも手元にハンドアウトをリスポーン
    - 公開済み情報・アイテムを確認できる機能
- テスト・デバッグ
    - 少人数でも進行確認できる開発者モード
    - 作品データの静的チェック
- 制作分担
    - ワールド共通基盤，脚本，アイテム素材を分担可能にする
    - 作品データとUnity実装を分離
    - アイテムは汎用モデルとテキスト情報で代替可能にする

## 課題定義

### 解決したい中核課題
- VRマダミスを遊ぶまでの人数・時間・制作負荷を下げる
- 一作ごとに専用ワールドを作り込む方式から，共通基盤に作品データを差し替える方式へ移行する
- Publicインスタンスでも参加・待機・観戦・感想戦が自然につながる場を作る
- ワールド内で遊ぶだけでなく，凝った作品や他制作者のワールドへのポータル集の役割も持ったポータルワールド化

### 設計上の問い
- VRChat内でどこまで作品データを差し替えられるか
- 作品ごとの人数条件をどう管理するか
- テーブル単位の進行状態をどう分離するか
- 観戦者にどこまで情報を見せるか
    - 観戦者には全ての公開情報・秘密情報を見せる
    - 観戦者は参加者とコミュニケーションが取れない(参加者目線からボイスミュート，アバター非表示)
- ユーザー制作作品の品質をどう担保するか
    - Git へのPRマージ前にポータルワールド製作者によるレビュー
    - レビューはガードレール的(公序良俗に反しないか程度)
        - ガイドライン整備は必要に応じて作成(初期スコープ外)

## コンセプト

### 一文コンセプト
- お手軽も，じっくりも，マダミス作品が集まったVRChatマダミスポータルワールド

### 体験コンセプト
- テーブルに集まって，手軽にマダミスが楽しめる
    - 来場者はロビーで作品カセットを眺める
    - 気になる作品のテーブルに参加する
    - 人数が揃ったらゲーム開始
    - 参加しない人は観戦エリアから状況を見られる
    - 終了後はそのまま感想戦に合流
- 参加者だけでなく，制作者にもなれる
    - 制作者はテンプレートに沿って作品データを作り，Git経由で追加候補にできる
- 様々なマダミスワールド作品を見つけられる
    - 既存のマダミスワールドへのポータル集を提供

## 想定ユーザーと利用シーン

### 想定ユーザー
- VRChatでマダミス・謎解き・会話ゲームを遊びたいユーザー
- 知り合いだけでなくPublicで遊びたいユーザー
- VRChat向けマダミス作品を作りたい制作者

### 利用シーン
- Publicインスタンスで集まった人が短編を遊ぶ
- イベント中に複数テーブルで同時進行する
- より凝ったVRマダミスを遊びたい人が作品を探す
- プレイヤーが観戦しながら次の参加枠を待つ
- 制作チームが新作カセットのテストプレイをする

## 作品ラインナップとプレイ人数モデル

### 初期ラインナップ目標
- 2人用作品を最低1本
- 3人用作品を最低1本
- 4人用作品を最低1本

### 人数設定
- 各カセットに以下を持たせる
    - `minPlayers`
    - `maxPlayers`
    - `recommendedPlayers`
    - 固定人数作品か人数可変作品か
- 初期MVPでは2人用，3人用，4人用の固定人数作品を優先
- 人数可変作品は拡張候補

### 途中参加
- プレイヤーとしての途中参加は不可
- 観戦者としての途中参加は可能
- 進行中テーブル近くの観戦エリアへ入ると状況が分かる
- 観戦エリアからの退出は自由

## プレイヤー体験フロー

1. ロビー入場
2. 作品カセット一覧を確認
3. 所要時間，人数，難易度，概要を確認
4. テーブルを選ぶ
5. 参加者が揃うまで待機
6. 役割・キャラクター決定
7. 共通開始ハンドアウト確認
8. 個別ハンドアウト確認
9. 議論
10. 投票
11. 結果発表
12. 共通終了ハンドアウト確認
13. 感想戦

## マッチング・待機・観戦フロー

### マッチング
- ワールド内に複数テーブルを配置
- 各テーブルでカセットを選択
- カセットに応じた必要人数を表示
- 参加者が揃ったテーブルから開始

### 待機
- 待機中に作品概要，所要時間，募集人数，現在参加者数を確認
- 参加待ちと観戦待ちを同じロビー導線に置く
- 人数不足時は別カセット選択や再募集へ戻れる

### 観戦
- 観戦エリアから進行中テーブルを見られる
- 途中から観戦可能
- 途中退出可能
- 観戦者には公開情報・秘密情報を含む全情報を見せる
- 観戦者は参加者とコミュニケーションできない
    - 参加者目線では観戦者ボイスをミュート
    - 参加者目線では観戦者アバターを非表示

## ポータルワールド構成

### 空間構成案
- ロビー
    - 本ワールド説明
    - 募集中テーブル一覧
- テーブルエリア
    - カセット棚
    - 作品説明パネル
    - 複数テーブル
    - テーブルごとの参加者スロット
    - カセット挿入口
    - 進行表示
- 観戦エリア
    - テーブルごとに近接配置
    - 観戦者用情報パネル
- 感想戦エリア
    - 終了後の合流場所
- 制作エリア
    - ユーザー入力によるマダミス制作のUI
    - 入力内容によるカセット生成
    - エクスポート用コード生成
    - インポート用UI (コード入力してカセット生成)
- ポータルエリア
    - カセット棚
    - 作品説明パネル
    - 他のマダミスワールドへのポータル出現スペース

### テーブル単位の独立性
- 各テーブルが独立した進行状態を持つ
- テーブルごとに参加者，カセット，フェーズ，投票結果を管理
- 複数テーブル同時稼働を将来目標とする
- MVPでは1テーブルから開始してもよい

- プレイ人数と所要時間の提示
- 共通基盤への作品読み込みトリガー
- 専用ワールドへの導線

### カセットに持たせる情報
- ゲーム名
- ゲーム概要
- プレイヤー人数
- 所要時間
- 難易度
- アイテム数
- ネタバレ・観戦設定
- 共通開始ハンドアウト
- 共通終了ハンドアウト
- プレイヤー設定
- アイテム設定
- 投票設定

### カセット種別
- ポータル内完結カセット
    - テーブルエリアに配置
    - 固定フェーズ構成
    - テキスト・画像・アイテム情報中心
    - 共通UIで進行
- 専用ワールドリンクカセット
    - ポータルエリアに配置
    - ワールド内移動や専用演出が必要
    - ポータルでは概要と導線のみ提供

    - メタデータはJSON
    - 本文はMarkdown

## ユーザー制作ゲーム取り込みフロー

### 初期方針
- 最初は非常にシンプルな制作範囲に限定
- 専用ギミック追加はMVPから外す
- フェーズ構成は固定
    - ハンドアウト確認
    - 議論
    - 投票
    - 発表

### 制作フロー案
1. 制作者が制作エリアに移動
2. ゲーム基本情報を記入
3. プレイヤー設定を記入
4. アイテム設定を記入
5. 投票設定を記入
6. 静的チェックを実行
7. カセット生成
8. テーブルエリアで遊べる
9. エクスポート用コードを発行してワールド制作者に連絡，またはPR作成
10. ワールド制作者がガイドライン観点レビューしてマージ
11. テーブルエリアのカセット棚に作品として並ぶ

### 品質担保
- 必須項目チェック
- 人数とプレイヤー設定数の整合性チェック
- アイテム数とアイテム設定数の整合性チェック
- 画像参照切れチェック
- 投票選択肢数チェック
- ネタバレ範囲確認

## MVP定義

### MVPで作るもの
- 1テーブル
- カセット選択
- 2人用，3人用，4人用カセット各1本
- 固定フェーズ構成
    - ハンドアウト確認
    - 議論
    - 投票
    - 発表
- 作品データのGit分離
- ハンドアウトのリスポーン
- 公開済み情報・アイテム確認
- 観戦エリア
- 開発者モード
- 作品データ静的チェック
- ポータルエリア，制作エリアの予定地
    - 空間上の配置と説明のみ
    - 実際のポータル機能・制作UIは実装しない

### MVPで作らないもの
- 専用ギミック追加
- 複雑なフェーズ編集
- 観戦者の高度な音声聞き分け
- 複数テーブル同時稼働
- 人数可変作品の自動役職調整
- ポータルエリアの実機能
- 制作エリアの実機能

## 技術制約

### VRChat制約
- UdonSharp / VRChat SDK上で動作
- 同期と所有権の扱いに注意
- 外部通信や動的読み込みには制約があるため，実現方式は検証が必要

### QvPen
- OSSとして活用
- 自前実装しない
- ポータル共通基盤との接続方法のみ検討

### 作品データ
- Git上では人が編集しやすい形式を優先
- Unity / UdonSharp側で扱いやすい形式への変換が必要な可能性
- VRChatワールド内での動的差し替え可否は要検証

## リスクと検証計画

### リスク
- カセット差し替えがVRChat上で想定通り実現できない
- テーブル単位の状態管理が複雑化する
- Publicマッチングで期待通り人が集まらない
- 初期作品数が少ないとポータルとしての魅力が弱い

### 検証項目
- 作品データをカセット単位で切り替えられるか
- 1テーブルで人数別カセットを開始できるか
- 2人用，3人用，4人用の固定人数作品が成立するか
- 観戦エリアから途中参加・途中退出できるか
- 作品データの静的チェックが実用的か
- 開発者モードで少人数進行確認できるか
